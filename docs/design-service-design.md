# 服务设计与体验蓝图

## 一句话总结
**服务设计（Service Design）把设计师的角色从"画一个屏幕"升维到"编排一段完整体验"**——从用户首次意识到问题，到任务完成、情绪落定的全链路。配合 Journey Map（用户视角情绪曲线）+ Service Blueprint（前后台分层）+ Double Diamond（双钻模型）三件套，你才能看见 UI 屏幕之间那些"看不到的"黑洞。

## 一、服务设计基础

**起源**：G. Lynn Shostack 1982 首次提出 "service design" 术语，1984《Harvard Business Review》"Breaking Free from Product Marketing" 奠定 Service Blueprint 理论基础。Vandermerwe & Rada 1988 系统化，2004 Service Design Network 在柏林成立。

**5 大原则**（Stickdorn & Schneider《This Is Service Design Thinking》综合版）：
1. **User-centered（以用户为中心）**——从用户需求而非业务 KPI 出发
2. **Co-creative（共同创造）**——用户、员工、第三方共同参与设计
3. **Sequencing（序列化）**——把无形的服务拆解成可视化、可被感知的环节
4. **Evidencing（物证化）**——用物理或数字证据让"看不见"的服务被"看见"（订单确认页、进度条）
5. **Holistic（整体观）**——同时考虑前台用户体验与后台支持流程

**关键区别**：服务设计 ≠ UX 设计。UX 通常聚焦数字产品某个屏幕的可用性；服务设计把视野扩展到产品之外——人、流程、基础设施、政策、物理环境。

## 二、Customer Journey Map（用户视角）

**NN/g 5 步法**：Define goal → Choose actor → Build timeline → Add thoughts/emotions → Visualize

**5 大组件**：
1. **Actor（用户画像）**——一张图只服务一个 persona，否则失焦
2. **Scenario + Expectations**——必须明确起点和终点
3. **Journey Phases**——发现 → 进入 → 使用 → 离开 / 复购
4. **Actions / Mindsets / Emotions**——尤其情绪曲线，能一眼看出"低谷"
5. **Opportunities**——每列痛点对应一个"如果……就好了"

**4Es 框架**（更适合长周期、高客单价服务如医疗/保险）：Exploration → Expectation → Experience → Evaluation。

**痛点不是 bug**，是"用户预期 vs 实际体验"的差距；**机会点不是功能**，是"如果我们重新设计这一段，会发生什么"。

## 三、Service Blueprint（让后台流程被看见）

**Shostack 1984 提出**，4 层：
1. **Customer Actions**——用户做了什么
2. **Front-stage Actions**——用户能看到的员工/系统响应
3. **Back-stage Actions**——用户看不到但触发了的员工/系统
4. **Support Processes**——IT、HR、合规、供应链等底层支撑

**3 条关键分隔线**：
- **Line of Interaction**——用户与系统的分界
- **Line of Visibility**——前台 vs 后台（决定哪些被"看见"）
- **Line of Internal Interaction**——直接服务部门 vs 支持部门

**最大差别**：Journey Map 看"用户的感受"，Blueprint 看"组织的责任"。痛点出现时，Journey 告诉你在哪一段，Blueprint 告诉你是哪一层哪个角色失职。

## 四、Double Diamond（双钻模型）

**英国 Design Council 2004/2005 正式发布**，两次"发散—收敛"：
- **Discover**——广角扫描问题，通过调研、访谈、共情图建立全貌
- **Define**——把发现收敛成一个清晰的"问题陈述"（How Might We…）
- **Develop**——再次发散，生成大量方案，原型化
- **Deliver**——收敛到可落地的最小可行方案，迭代上线

**4 条核心原则**：Put people first / Communicate visually and inclusively / Collaborate and co-create / Iterate, iterate, iterate

**注意**：Double Diamond 不是瀑布流——Develop 阶段可能跳回 Define，这是常态而非失控。

## 五、Service Prototyping

服务看不见摸不着，"画出来"不够，必须"演出来"。三种方法：
1. **Role-play（角色扮演）**——团队成员扮演用户/前台/后台，在白板前走一遍剧本
2. **Video prototyping**——拍 1-3 分钟短片，让抽象服务"被看见"
3. **Desktop walkthrough**——纸片、便签、卡片在桌面上"演"完一次完整服务

**真实案例**：Airbnb 早期大量用 role-play 验证"房东—客人"双向流程；GOV.UK 重塑签证服务时，先用视频短片测试 100 多种"用户试图解释自己情况"的真实对话。

## 六、Touchpoint（接触点）

**Wikipedia 7 类触点**：atmospheric（氛围）/ technological（技术）/ communicative（沟通）/ process（流程）/ employee-customer（员工—客户）/ customer-customer（客户—客户）/ product interaction（产品交互）

**按归因分**：
- **公司创造型**——广告、官网、APP
- **内在型**——产品本身、门店
- **意外型**——社交媒体、新闻报道
- **客户自发型**——用户评论、社群

**触点不是孤立的点，而是一条时间链**。一个用户在医院看病的触点链至少 12 个：搜索症状 → 选医院 → 预约 → 停车 → 导诊 → 候诊 → 问诊 → 检查 → 复诊 → 缴费 → 取药 → 随访。**任何一个断点都会毁掉全链体验**。

## 七、真实企业案例的关键决策

1. **IKEA**——把"动线"作为产品本身设计：顾客必须穿过所有展示区；"自己组装"作为用户参与设计的一部分
2. **Airbnb**（2009 濒临破产时）——用 Craigslist 寄生流量 + 邀请信任机制 + 双向评分系统，把"陌生人住进陌生人家"变成"被设计过的信任流程"——**这是服务设计而非产品设计**
3. **Mayo Clinic**——把"患者在 17 个科室间被踢来踢去"重组成"主诊医生 + 集中式检查路线 + 1 对 1 病例协调员"的一站式服务，把 3 周就诊流程压缩到 3 天
4. **GOV.UK**——把 800+ 个政府网站整合成 1 个入口 + 1 套 Service Standard，每项服务必须通过 14 条标准（如"用用户能听懂的语言""持续做用户研究"），违者下架
5. **传统银行**——手机银行让"排队"消失，但后台客服/信贷/合规流程没改，结果用户线上申请到放款仍是 7 天——**典型的"前台优化、后台失配"**
6. **医院数字化**——预约挂号上线，但患者到院后仍要重新填表，触点断裂，NPS 反而下降
7. **瑞银 UBS**——在私人银行业务中引入"客户旅程总监"角色，与产品经理平级
8. **荷兰皇家航空 KLM**——在 Twitter 7×24 真人 1 对 1 回应，把社交媒体变成"情感触点"

## 八、服务设计工具

- **Miro / Mural**——在线白板，适合远程团队画 Journey Map 和 Blueprint
- **Smaply / Custellence**——专业服务设计 SaaS，模板化 Journey Map / Blueprint / Persona / Stakeholder Map
- **ServiceDesignTools.org**——开源工具库，47 种方法按 "When / Who / What / How" 四维分类
- **Figma / FigJam**——原型化数字触点
- **纸 + 笔 + 便利贴**——永远是最低成本、最好用的服务设计工具

## 画 Service Blueprint 的 8 步骤

1. **明确目标场景**——选定一个具体用户场景（如"老用户续费"），不要试图一张图画完整业务
2. **列出客户行为**——从用户视角，列出他在每个阶段做什么（搜索、点击、打电话、等候）
3. **画出 Line of Visibility**——明确"用户能看到什么"作为分界
4. **列出前台交互**——用户能看到的员工动作、系统响应（弹窗、邮件、SMS）
5. **列出后台动作**——用户看不到但被触发的流程（订单同步、库存扣减、审核）
6. **列出支持流程**——IT、HR、合规、供应商等底层系统
7. **标记 fail point**——在每层之间标注"如果 X 失败会怎样"
8. **指派责任人**——每个动作必须有 owner，否则蓝图只是装饰品

## 从 UX 升维到服务设计的 7 个动作

1. **从屏幕到旅程**——把单页设计的 review 改为"端到端 journey review"
2. **从用户到生态**——除了用户，把前台员工、后台员工、第三方都画进 Stakeholder Map
3. **从可用性到情感**——每个触点都画情绪曲线，找到低谷
4. **从功能到流程**——每个功能都问"前后台谁在执行？"
5. **从原型到演戏**——用 role-play 把服务演出来，而不是只做 click-through
6. **从指标到体验**——NPS、CSAT 是结果指标，体验旅程的"过程指标"才是服务设计的发力点
7. **从设计交付到组织设计**——当 Blueprint 暴露组织问题（如"客服无权退款"）时，勇敢推动组织改动

## 3 条"如果你只能记一条"

1. **不要只画界面，画整段"客户从发现到完成"的人生**。UI 是冰山露出水面的尖——Blueprint 才是水面下的 90%。当你画完第一张 Service Blueprint，你会发现你设计的不是产品，而是一家公司与用户之间的整段关系
2. **Journey Map 告诉你痛点在哪一段，Blueprint 告诉你是哪一层哪个角色失职**——两个图一起画才是完整服务诊断
3. **前台优化 + 后台失配 = 用户体验更差**（手机银行让排队消失但放款还是 7 天）——服务设计永远同时看前后台
